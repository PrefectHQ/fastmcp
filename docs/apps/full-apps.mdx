Full apps go beyond single-screen tool UIs. They combine multiple pages, shared state, and server-side actions into cohesive applications — all rendered inside the host's conversation.

<Info>
  This section is under active development. The patterns described here work today, but we're building higher-level abstractions to make multi-page apps even easier.
</Info>

## Pages

The `Pages` component provides client-side navigation between views within a single tool result. Each `Page` is a named section, and only the active page renders at a time:

```python
from prefab_ui.components import *
from prefab_ui.app import PrefabApp
from fastmcp import FastMCP

mcp = FastMCP("CRM")

@mcp.tool(app=True)
def crm_dashboard() -> PrefabApp:
    """Full CRM dashboard with multiple views."""
    contacts = fetch_contacts()
    deals = fetch_deals()

    with Column(css_class="p-6") as view:
        Heading("CRM Dashboard")

        with Pages(default_value="contacts"):
            with Page("Contacts"):
                DataTable(
                    columns=[
                        DataTableColumn(key="name", header="Name", sortable=True),
                        DataTableColumn(key="email", header="Email"),
                        DataTableColumn(key="company", header="Company", sortable=True),
                    ],
                    rows=contacts,
                    searchable=True,
                    paginated=True,
                )

            with Page("Deals"):
                DataTable(
                    columns=[
                        DataTableColumn(key="title", header="Deal", sortable=True),
                        DataTableColumn(key="value", header="Value", sortable=True),
                        DataTableColumn(key="stage", header="Stage"),
                    ],
                    rows=deals,
                    searchable=True,
                )

            with Page("New Contact"):
                with Form(on_submit=ToolCall("create_contact")):
                    Input(name="name", label="Name", required=True)
                    Input(name="email", label="Email", input_type="email")
                    Input(name="company", label="Company")
                    Button("Create")

    return PrefabApp(
        view=view,
        state={"contacts": contacts, "deals": deals},
    )
```

Pages render a navigation bar automatically. The user clicks between views without any server round-trip — the page switch is pure client-side state.

## Shared State Across Pages

State declared in `PrefabApp(state={...})` is available to all pages. This means a form on one page can write state that a table on another page reads:

```python
@mcp.tool(app=True)
def inventory_manager() -> PrefabApp:
    """Manage inventory with add form and list view."""
    items = fetch_inventory()

    with Column(css_class="p-6") as view:
        with Pages(default_value="list"):
            with Page("Inventory"):
                DataTable(
                    columns=[
                        DataTableColumn(key="name", header="Item", sortable=True),
                        DataTableColumn(key="quantity", header="Qty", sortable=True),
                        DataTableColumn(key="location", header="Location"),
                    ],
                    rows="{{ items }}",
                    searchable=True,
                )

            with Page("Add Item"):
                with Form(
                    on_submit=ToolCall(
                        "add_inventory_item",
                        result_key="items",
                        on_success=ShowToast("Item added!", variant="success"),
                    )
                ):
                    Input(name="name", label="Item Name", required=True)
                    Input(name="quantity", label="Quantity", input_type="number")
                    Input(name="location", label="Location")
                    Button("Add Item")

    return PrefabApp(view=view, state={"items": items})
```

The key here is `result_key="items"` on the `ToolCall`. When the server tool returns, its result replaces the `items` state — and the table on the other page automatically reflects the new data.

## Server Actions as the Backend

Full apps treat your MCP tools as a backend API. The initial tool renders the UI, and buttons and forms call other tools to mutate data:

```python
@mcp.tool(app=True)
def task_board() -> PrefabApp:
    """Interactive task board."""
    tasks = fetch_tasks()

    with Column(gap=4, css_class="p-6") as view:
        with Row(gap=2, align="center"):
            Heading("Tasks")
            Button(
                "Refresh",
                on_click=ToolCall("fetch_tasks", result_key="tasks"),
            )

        with ForEach("tasks"):
            with Card():
                with CardContent():
                    with Row(gap=2, align="center", justify="between"):
                        Text("{{ title }}", css_class="font-medium")
                        Badge("{{ status }}")
                    Muted("{{ description }}")

    return PrefabApp(view=view, state={"tasks": tasks})


@mcp.tool
def fetch_tasks() -> list[dict]:
    """Fetch all tasks from the database."""
    return db.get_tasks()
```

The `fetch_tasks` tool serves double duty: it's the backend for the refresh button, and it could be called directly by the model. This is one of the nice things about building on MCP — your UI actions and your conversational tools are the same thing.

## Coming Soon

We're working on patterns and components to make full apps more ergonomic:

- **Router-style navigation** — URL-like routing between pages with parameter passing
- **Optimistic updates** — Update state immediately, roll back on server error
- **Server-pushed state** — Stream state updates from long-running server operations

For now, the `Pages` + `ToolCall` + `result_key` pattern handles the vast majority of multi-view use cases. See the [Prefab UI documentation](https://prefab.prefect.io) for the complete component reference.
