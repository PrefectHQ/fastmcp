[Prefab UI](https://prefab.prefect.io) is a Python component library that gives your MCP tools rich, interactive UIs. You write Python ŌĆö Prefab renders charts, tables, forms, and layouts directly inside the host client's conversation. No HTML. No JavaScript. No frontend build step.

```bash
pip install "fastmcp[apps]"
```

## The Idea

Most MCP tools return text. That's fine for simple answers, but some results ŌĆö a sales chart, a sortable employee directory, a form that collects user input ŌĆö deserve better than a JSON blob in a chat window.

Prefab lets you describe those UIs in Python using composable components, and FastMCP handles everything else: registering the renderer, wiring up the protocol, and delivering the result to the host.

```python
from prefab_ui.components import Column, Heading, BarChart, ChartSeries, Muted
from prefab_ui.app import PrefabApp
from fastmcp import FastMCP

mcp = FastMCP("Dashboard")


@mcp.tool(app=True)
def revenue_chart(year: int) -> PrefabApp:
    """Show annual revenue as an interactive bar chart."""
    data = get_revenue_data(year)
    total = sum(d["revenue"] for d in data)

    with Column(gap=4, css_class="p-6") as view:
        Heading(f"{year} Revenue")
        Muted(f"${total:,} total")
        BarChart(
            data=data,
            series=[ChartSeries(data_key="revenue", label="Revenue")],
            x_axis="quarter",
        )

    return PrefabApp(view=view)
```

When a host calls `revenue_chart`, the user sees an interactive chart with hover tooltips and axis labels ŌĆö not a wall of numbers.

## How It Works

Setting `app=True` (or returning a `PrefabApp` type) tells FastMCP to:

1. **Register a shared renderer** ŌĆö a `ui://prefab/renderer.html` resource containing the Prefab rendering engine, fetched once and reused across tools
2. **Wire the tool metadata** ŌĆö so the host knows to load the renderer iframe when calling the tool
3. **Serialize the component tree** ŌĆö your Python components become `structuredContent` on the tool result, which the renderer interprets and displays

You don't manage resources, write HTML, or think about the MCP Apps protocol. You build a component tree and return it.

## Building UIs

Prefab uses Python context managers for layout composition. Components created inside a `with` block automatically become children of the parent:

```python
from prefab_ui.components import Column, Row, Card, CardContent, Heading, Text, Badge

with Column(gap=6, css_class="p-6") as view:
    with Row(gap=2, align="center"):
        Heading("Team Dashboard")
        Badge("Live", variant="success")

    with Card():
        with CardContent():
            Text("Everything is running smoothly.")
```

This produces a vertical layout with a header row (title + badge) and a card below it. The `gap`, `align`, and `css_class` parameters control spacing, alignment, and styling using Tailwind CSS classes.

## Return Patterns

There are three ways to return UI from a tool, depending on how much control you need:

### Return a PrefabApp

The standard pattern. Attach client-side state when your UI needs it:

```python
@mcp.tool(app=True)
def team_directory() -> PrefabApp:
    members = get_members()

    with Column(gap=4, css_class="p-6") as view:
        Heading("Team")
        DataTable(
            columns=[
                DataTableColumn(key="name", header="Name", sortable=True),
                DataTableColumn(key="role", header="Role", sortable=True),
            ],
            rows=members,
            searchable=True,
        )

    return PrefabApp(view=view, state={"count": len(members)})
```

### Return a Bare Component

For simple cases where you don't need state. FastMCP wraps it automatically:

```python
@mcp.tool(app=True)
def status_badge() -> Column:
    with Column(gap=2) as view:
        Heading("All Systems Operational")
        Badge("Healthy", variant="success")
    return view
```

### Custom Text Fallback

By default, LLM clients that don't render Apps see `"[Rendered Prefab UI]"`. To provide a more useful summary the model can reason about, wrap the result in a `ToolResult`:

```python
from fastmcp.tools import ToolResult

@mcp.tool(app=True)
def sales_overview() -> ToolResult:
    total = sum(row["revenue"] for row in data)

    with Column(gap=4, css_class="p-6") as view:
        Heading("Sales Overview")
        BarChart(data=data, series=[ChartSeries(data_key="revenue")])

    return ToolResult(
        content=f"Total revenue: ${total:,}",
        structured_content=view,
    )
```

The `content` string is what the LLM sees; the `structured_content` is what the renderer displays.

## Type Inference

You don't strictly need `app=True` if your return type annotation is a Prefab type. FastMCP detects `PrefabApp`, `Component`, and their `Optional` variants automatically:

```python
# app=True is inferred from the return type
@mcp.tool
def greet(name: str) -> PrefabApp:
    return PrefabApp(view=Heading(f"Hello, {name}!"))
```

Explicit `app=True` is still recommended for clarity. It's required when the return type doesn't reveal the Prefab type (e.g., `-> ToolResult`).

## Interactivity

Prefab UIs aren't static renders ŌĆö they support client-side state, form inputs, and actions that call back to your server.

### Client-Side State

State values are set in `PrefabApp(state={...})` and referenced in components using `{{ expression }}` templates:

```python
from prefab_ui.components import Column, Text, Button, If, Badge
from prefab_ui.actions import ToggleState

@mcp.tool(app=True)
def toggle_demo() -> PrefabApp:
    with Column(gap=4) as view:
        Button("Toggle", on_click=ToggleState("show"))
        with If("{{ show }}"):
            Badge("Visible!", variant="success")

    return PrefabApp(view=view, state={"show": False})
```

State mutations happen entirely in the browser ŌĆö no server round-trip needed for `SetState`, `ToggleState`, `AppendState`, or `PopState`.

### Server Actions

Buttons and forms can call tools on your MCP server:

```python
from prefab_ui.actions import ToolCall, ShowToast

Button(
    "Save",
    on_click=ToolCall(
        "save_record",
        arguments={"name": "{{ name }}", "email": "{{ email }}"},
        on_success=ShowToast("Saved!", variant="success"),
        on_error=ShowToast("{{ $error }}", variant="error"),
    ),
)
```

The renderer executes the tool call, handles the response, and runs success/error callbacks ŌĆö all declared in Python.

### Forms

Named inputs automatically sync their values to client-side state. The `Form` component collects them on submit:

```python
from prefab_ui.components import Form, Input, Button
from prefab_ui.actions import ToolCall

with Form(on_submit=ToolCall("create_user")):
    Input(name="username", placeholder="Username", required=True)
    Input(name="email", input_type="email", placeholder="Email")
    Button("Create Account")
```

For Pydantic models, `Form.from_model()` auto-generates the entire form ŌĆö inputs, labels, validation, and types:

```python
from pydantic import BaseModel, Field

class UserProfile(BaseModel):
    name: str = Field(title="Full Name")
    email: str = Field(title="Email")
    role: Literal["user", "admin"] = "user"

Form.from_model(UserProfile, on_submit=ToolCall("save_profile"))
```

## Mixing with the Low-Level API

Prefab tools and custom HTML tools coexist in the same server:

```python
from fastmcp.server.apps import AppConfig

# Prefab tool ŌĆö automatic renderer
@mcp.tool(app=True)
def team_directory() -> PrefabApp:
    ...

# Custom HTML tool ŌĆö your own renderer
@mcp.tool(app=AppConfig(resource_uri="ui://my-app/map.html"))
def map_view() -> str:
    ...
```

Prefab tools share a single renderer resource. Custom tools point to their own. Both use the same MCP Apps protocol under the hood.
